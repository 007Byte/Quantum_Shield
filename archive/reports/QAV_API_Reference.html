The QAV API provides secure, post-quantum encrypted file storage with zero-knowledge architecture. All file content is encrypted client-side before transmission. The server never has access to plaintext data or master keys. Authentication uses SRP-6a (zero-knowledge password proof) or FIDO2/WebAuthn for passwordless access.

🔐

Authentication

🔒
All POST /api/v1/auth/srp/verify and FIDO2 verify responses return an access_token (15-min TTL) and refresh_token (7-day TTL). Include the access token as Authorization: Bearer <token> on all authenticated endpoints.

SRP-6a Flow

1. POST /auth/srp/init — send email, receive salt + B
2. Compute A, M1 client-side using SRP-6a
3. POST /auth/srp/verify — send A + M1, receive JWT tokens
Password is never transmitted or stored.

JWT Key Rotation (PH2-FIX)

Access tokens include a kid header identifying the signing key version. Multiple active keys are supported during rotation windows. Admin-only endpoint POST /api/v1/admin/rotate-jwt-keys triggers graceful key rotation.

⚠️

Error Handling

All error responses follow a standardized structure:

"error": "VAULT_NOT_FOUND", "message": "Vault not found or access denied", "request_id": "req-abc123" // for support tracing
Common Error Codes
UNAUTHORIZED — 401 · Invalid/expired token
FORBIDDEN — 403 · Insufficient permissions
VAULT_NOT_FOUND — 404 · Vault missing or denied
VALIDATION_ERROR — 400 · Malformed request
RATE_LIMITED — 429 · Too many requests
TIER_LIMIT_EXCEEDED — 403 · Plan limit reached
ROTATION_IN_PROGRESS — 409 · Key rotation active
Rate Limits
Auth endpoints: 10 req/min per IP
SRP init/verify: 5 req/min per email
Vault operations: 60 req/min per user
Blob upload/download: 30 req/min
Admin endpoints: admin role required

POST /api/v1/auth/fido2/challenge Get WebAuthn challenge for passwordless login Public ▶

Description

Returns a WebAuthn challenge for FIDO2 assertion. The client uses the challenge with a registered authenticator (hardware key, Touch ID, Face ID) to generate a signed assertion.

POST /api/v1/auth/fido2/verify Verify WebAuthn assertion — returns JWT tokens Public ▶

Verifies the authenticator assertion. On success returns the same access/refresh token pair as SRP verify.

POST /api/v1/auth/fido2/manage/register/init Initialize FIDO2 credential registration Bearer JWT ▶

Returns FIDO2RegisterInitResponse with WebAuthn creation options. Pass these to navigator.credentials.create() to generate the credential.

POST /api/v1/auth/fido2/manage/register/verify Verify FIDO2 credential registration Bearer JWT ▶

Completes credential registration. Stores the new credential linked to the authenticated user account.

GET /api/v1/auth/fido2/manage/credentials List registered FIDO2 credentials Bearer JWT ▶

Returns all registered FIDO2 credentials for the user. Also supports DELETE on /credentials/{credentialID} to deregister a specific device.

Key Rotation

2 endpoints
POST /api/v1/vaults/{vaultID}/rotate Initiate vault key rotation (vault owner only) Bearer JWT · Owner ▶

Description

Initiates asynchronous key rotation. All vault blobs will be re-encrypted with a new vault key. During rotation, the vault remains accessible using the current key. Returns rotation_id for tracking progress. Returns 409 if rotation is already in progress.

GET /api/v1/vaults/{vaultID}/rotation-status Get key rotation progress Bearer JWT ▶

Returns RotationStatusResponse with status (pending/in_progress/complete/failed), progress_pct, files_rotated, total_files, and started_at.

Blobs

4 endpoints · presigned S3/MinIO URLs
📤
QAV uses presigned URLs for all blob transfers — the app never proxies file data through the API server. Files are encrypted client-side before upload. The API provides short-lived presigned URLs to upload/download directly to/from MinIO (or S3 in production).
GET /api/v1/vaults/{vaultID}/blobs List blobs in vault Bearer JWT ▶

Returns paginated list of blob metadata. File content is not returned — use download-url endpoint to obtain a presigned URL for retrieval.

Returns a short-lived presigned URL to download the encrypted blob directly from storage. The client must decrypt after download using the vault key.

Sharing

8 endpoints
🔒
P2P shares use X25519 sealed-box encryption — only the recipient can decrypt. Key fingerprints should be verified out-of-band (via /shares/verify-contact or /shares/fingerprint/{userID}) before sharing sensitive files.
POST /api/v1/shares Create a share (P2P encrypted file share) Bearer JWT ▶

Request Body

FieldTypeDescription
recipient_user_idstring (uuid)Target user
blob_idstring (uuid)Blob to share
sealed_share_keystring (base64)File key sealed with recipient's X25519 public key
expires_atstring (ISO 8601)optional

Responses

201Share created. Returns share_id and metadata.
404Recipient not found or no public key registered.
GET /api/v1/shares/sent | /api/v1/shares/received List sent or received shares Bearer JWT ▶

Returns paginated list of shares. Includes status (pending / accepted / rejected / revoked / expired).

POST /api/v1/shares/{shareID}/accept Accept a received share Bearer JWT ▶

Marks share as accepted. After accepting, use POST /vaults/{id}/blobs/download-url to retrieve the blob, then decrypt using the sealed_share_key with your X25519 private key.

GET /api/v1/shares/public-key/{userID} Get a user's X25519 public key for encryption Bearer JWT ▶

Returns the user's registered X25519 public key (hex encoded). Use this to encrypt the share key before calling POST /shares. Always verify the fingerprint out-of-band before trusting for sensitive shares.

Audit

5 endpoints
🔗
QAV uses a tamper-evident blockchain-style audit log. Each entry is hashed with the previous entry's hash, creating an immutable chain. The /verify endpoint validates chain integrity.
GET /api/v1/audit List audit log entries Bearer JWT ▶

Query Parameters

ParamTypeDescription
fromstring (ISO 8601)Start of time range
tostring (ISO 8601)End of time range
event_typestringFilter by event type (e.g., vault.access, file.download)
vault_idstring (uuid)Filter to specific vault
GET /api/v1/audit/verify Verify audit chain integrity Bearer JWT ▶

Re-validates the entire audit chain hash sequence. Returns valid: true if chain is intact, or identifies the first tampered entry if chain is broken.

GET /api/v1/audit/anomalies Get detected security anomalies Bearer JWT ▶

Returns anomalies detected by the real-time security monitor: unusual access patterns, brute-force attempts, off-hours access, bulk download events.

GET /api/v1/audit/compliance-report Generate compliance report Bearer JWT · Admin ▶

Generates a structured compliance report for the requested time period. Suitable for SOC 2, GDPR, and HIPAA audit evidence. Use GET /audit/compliance-export (Pro tier required) to export as CSV.

Admin-only. Triggers graceful JWT key rotation: generates new signing key, keeps old key active for in-flight token validation during TTL window. Returns new kid (key ID). Part of PH2-FIX zero-downtime key rotation.

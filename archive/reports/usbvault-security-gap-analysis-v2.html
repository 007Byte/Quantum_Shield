Security Architecture Gap Analysis — v2

USBVault Enterprise · Updated March 12, 2026 · Delta from v1 (March 8, 2026)

Bottom-Line Assessment — Significantly Improved

Since v1 (March 8), 7 of the 8 critical and high gaps have been fully resolved. The two-layer key hierarchy (KEK + MEK), per-file HKDF keys, recovery codes, vault index encryption, FIDO2, SRP Argon2id verifier migration, key change notifications, and rollback protection are all now implemented and verified in code.

The core cryptographic architecture is now production-grade: Argon2id KDF → KEK → random MEK → per-file HKDF keys, with XChaCha20-Poly1305/AES-256-GCM-SIV AEAD, encrypted vault index, monotonic version counters, recovery codes, and full FIDO2/WebAuthn.

Remaining gaps are operational (not architectural): PQC is available but not integrated into the main key path, certificate pinning needs production pins, and the forensics service is partially stubbed. The "zero-knowledge" claim is now substantially more accurate — file names are encrypted in the vault index, and a metadata reduction service handles timing/size obfuscation.

01 Post-Quantum Cryptography (PQC) Critical Gap Improved ▾
Progress Since v1

ML-KEM-1024 is now available as an optional Cargo feature (pqc feature flag in Cargo.toml) with FFI integration harness tests for keypair generation, seal, and open. However, it is not yet integrated into the main key hierarchy.

Remaining Gap

The pqc feature is opt-in and not wired into production key exchange. X25519 is still the sole KEM for sharing. No ML-DSA (Dilithium) for signatures. CNSA 2.0 compliance document claims should still be qualified until hybrid KEM is the default path.

Cargo.tomlLine 53: ML-KEM-1024 dependency (optional). Line 63: pqc = [...] feature flag defined
FFI Testsffi_integration_harness_test.rs: ML-KEM keypair, seal/open operations tested
ProductionFeature flag not enabled by default — hybrid KEM not in main key path
Remaining Action

Enable the pqc feature flag and wire hybrid X25519 + ML-KEM-768 into the sharing sealed-box flow. Combine shared secrets via HKDF. Add ML-DSA-65 for signature operations. Until then, qualify CNSA 2.0 claims as "PQC-ready" rather than "PQC-compliant."

02 Key Management & Key Hierarchy Resolved Fixed ▾
Fully Resolved Since v1

The two-layer key hierarchy recommended in v1 is now implemented in keyHierarchy.ts. Password → Argon2id → KEK (wrapping key, 32 bytes) → wraps random MEK (64 bytes) → HKDF per-file keys. Password change is O(1) — only re-wraps MEK.

03 Metadata Leakage & "Zero-Knowledge" Claim Medium Gap Improved ▾
Major Improvement Since v1

The vault index (file names + metadata) is now encrypted client-side with XChaCha20-Poly1305 using a key derived via HKDF(master_key, "vault_index_encryption"). A metadata reduction service provides timing jitter, message padding, and EXIF stripping. The "zero-knowledge" claim is now substantially more accurate.

Operational metadata (sync event timing, device sessions) remains visible to the server — this is inherent in any client-server architecture. Timing jitter and batch delivery are configurable but disabled by default. Consider enabling by default in enterprise tier. ORAM remains impractical for the current architecture.

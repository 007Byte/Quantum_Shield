Security Architecture Gap Analysis

Quantum Armor Vault (QAV) · Code Audit Against External Security Review · March 8, 2026

Bottom-Line Assessment

The core cryptographic primitives are solid — Argon2id KDF, XChaCha20-Poly1305/AES-256-GCM-SIV AEAD, HMAC-SHA256 integrity, X25519 key exchange, SRP-6a zero-knowledge auth. The Rust implementation includes proper memory zeroing, constant-time comparisons, and fuzz testing.

However, there are real gaps between documentation claims and code reality. The CNSA 2.0 compliance document claims ML-KEM-1024 and ML-DSA-87 implementations that do not exist in the Rust crypto core. The "zero-knowledge" claim is technically inaccurate — file names are stored as plaintext HashMap keys in the vault index, and the server sees sharing graphs, file sizes, and access patterns.

The architecture is strongest as: an encrypted-storage system with strong client-side cryptography and a privacy-respecting server. It is weakest as: a literal "zero-knowledge" system where the server learns nothing about user activity.

01 Post-Quantum Cryptography (PQC) Critical Gap ▾
Documentation vs. Reality Mismatch

CNSA_2_0_COMPLIANCE.md claims "ML-KEM-1024 implementation" and "ML-DSA-87 (CRYSTALS-Dilithium)" in vault_pqc.py. However, vault_pqc.py is a Python module in the legacy codebase, not in the Rust Enterprise crypto core.

Cargo.tomlLine 59: pqc = [] # Post-quantum crypto (future: oqs crate) — Feature flag is EMPTY, no crate dependency
Rust SourceZero references to ML-KEM, Kyber, Dilithium, or any PQC algorithm across all 16 .rs files
X25519sharing.rs uses X25519 alone — NOT hybridized with any PQC KEM
SignaturesEd25519 only — no ML-DSA/Dilithium/Falcon
Recommendation

Implement hybrid KEM: X25519 + ML-KEM-768 (FIPS 203) using the oqs or pqcrypto Rust crate. Combine shared secrets via HKDF as described in your own CNSA 2.0 doc (which is correct in theory, just not implemented). Until then, retract PQC claims from documentation to avoid misleading security auditors.

02 Key Management & Key Hierarchy High Gap ▾
PropertyStatusEvidence
Master key generationPassword-derivedkdf.rs: Argon2id(65MB, 3 iter, 4 lanes) → 64 bytes split into enc_key + hmac_key
Key Encryption Key (KEK)MissingNo separate random master key wrapped by a password-derived KEK. Password directly determines all data keys.
Hardware security (TPM/Enclave)MissingRust core has no hardware integration. Only software mlock/munlock in memory.rs
Secure memory (zeroize)ImplementedZeroizing<Vec<u8>> wrapper on all key material
Memory locking (mlock)Implementedmemory.rs: platform-specific mlock for Linux/macOS/Windows
Weakness: Password = Single Point of Failure

The current design derives all vault keys directly from the password via Argon2id. A weak password compromises ALL data. Best practice is to generate a random master key, then wrap it with a password-derived key. This allows password changes without re-encrypting all data and enables future hardware key storage.

Recommendation

Introduce a two-layer key hierarchy: (1) Random 256-bit Master Key generated at account creation, (2) Password-derived Key Encryption Key (KEK) via Argon2id that wraps the Master Key. Store the wrapped master key alongside the salt. On password change, only re-wrap — no data re-encryption needed. This also enables future TPM/Secure Enclave backing of the unwrapped master key.

03 Metadata Leakage & "Zero-Knowledge" Claim Critical Gap ▾
The "Zero-Knowledge" Claim Is Overstated

The server cannot read file contents, but it knows substantially more than "nothing" about user activity.

Metadata TypeStatusEvidence
File contentsEncryptedXChaCha20-Poly1305 or AES-256-GCM-SIV AEAD
File namesPLAINTEXTvault/index.rs: HashMap<String, u64> — filenames are literal keys serialized as JSON
File sizesVisibleNo padding applied. Encrypted blob size ≈ plaintext size + AEAD overhead
Access patternsVisibleS3 presigned URLs logged server-side. No ORAM or access obfuscation
Upload/download timingVisibleServer issues presigned URLs and sees all request timestamps
Sharing graph (who → whom)VisibleServer manages share records in PostgreSQL with sender/recipient IDs
Account existenceProtectedSRP has dummy computation + random delay for non-existent accounts
Device listVisibledeviceID passed to server as string
Number of filesVisibleVault CRUD exposes object count
Recommendation

Immediate: Change marketing language from "zero-knowledge" to "end-to-end encrypted with limited server knowledge" or "encrypted storage with privacy-respecting server." Short-term: Encrypt the vault index (file names + offsets) client-side before storing server-side. Long-term: Pad file sizes to fixed buckets, encrypt sharing metadata, and evaluate ORAM for access pattern obfuscation (high-latency trade-off).

Add a monotonic version counter per file entry, authenticated in the AEAD additional data. This prevents the server from silently rolling back a file to an older encrypted version. Also expand domain separation with unique HKDF context strings for each key derivation path.

06 SRP-6a Implementation Medium Gap ▾
PropertyStatusEvidence
LibraryCustomManual BigInt via num-bigint 0.4 (Rust) and custom srp.go (Go)
Group parametersRFC 50543072-bit group (RFC 7919), hardcoded in both implementations
Parameter validationImplementedServer: B ≠ 0 mod N; Client: 1 < A < N; k = H(PAD(N) || PAD(g))
Account enumerationProtectedDummy computation + random 0–100ms delay + constant-time comparison
Session binding to TLSMissingNo RFC 5929 channel binding. SRP proof not tied to transport
Password hash for verifierSHA onlySRP verifier uses x = H(salt, H(identity:password)) — Argon2id used only for vault KDF, NOT for SRP password hash
SRP Verifier Weakness

The SRP verifier is derived from a simple SHA hash of the password, not from Argon2id. If the verifier database is stolen, an attacker can brute-force weak passwords much faster than if Argon2id were used. This is a common oversight — the expensive KDF protects the vault keys but not the authentication credential.

Recommendation

Use Argon2id as the inner hash for SRP verifier generation: x = Argon2id(password, salt) → SRP verifier. This makes offline brute-force against stolen verifiers as expensive as attacking the vault keys. Also add TLS channel binding (tls-server-end-point) to prevent SRP session relay attacks.

08 Recovery, Reset & Password Change High Gap ▾
PropertyStatusEvidence
Password recovery mechanismNone (by design)Forgotten password = permanent data loss
Key escrowNone (Good)Server never holds master keys — true zero-knowledge for key material
Recovery codesMissingNo backup recovery code system
Password change → data re-encryptionMissingSRP verifier updates but vault data is NOT re-encrypted with new key
Usability vs. Security Trade-off

The "no recovery" design is maximally secure but will cause real data loss for real users. Enterprise customers typically require some recovery mechanism. Without one, a single forgotten password means losing all corporate encrypted data.

Recommendation

Implement recovery codes: at account creation, generate a set of one-time codes that can unwrap the master key (this requires the two-layer key hierarchy from Section 02). These codes are displayed once, never stored on the server. For enterprise, consider organizational key escrow where a company admin holds a recovery share (Shamir's Secret Sharing). This preserves the zero-knowledge model while enabling business continuity.

09 Forward Secrecy & Key Rotation High Gap ▾
PropertyStatusEvidence
Per-file encryption keysMissingAll files encrypted with same master-derived key — no per-file key rotation
Share revocation re-encryptionMissingRevokeShare() removes server record only. Revoked user retains ciphertext
Double Ratchet for messagingMissingNo ratchet protocol found
Key rotation on password changeMissingPassword change updates SRP verifier only, not vault keys
Ephemeral keys for sharingImplementedX25519 ephemeral keypair per share operation
Recommendation

Per-file keys: Generate a random file encryption key (FEK) per file, encrypt the FEK with the master key, store wrapped FEK alongside the blob. Revocation: On share revocation, re-encrypt the file with a new FEK and re-wrap for remaining authorized users. Messaging: If the Messages feature becomes a real E2E messaging system, implement Signal's Double Ratchet for forward secrecy and post-compromise security.

11 Client Integrity & Update Model Medium Gap ▾
Good newsNo web client — the app is native React Native only. The server cannot serve malicious JavaScript.
Bad newsNo code signing verification, no reproducible builds, no remote attestation.

The reviewer's concern about "serve a malicious client update that exfiltrates keys" is valid. App store distribution provides some protection (Apple/Google review), but a compromised build pipeline or sideloaded APK could bypass this. The appProtection.ts and deviceIntegrity.ts services do basic integrity checks but don't verify the app binary itself.

Recommendation

Implement binary transparency: publish build hashes to a public log. Add iOS App Attest for the crypto operations and Android Play Integrity API. For enterprise distribution, implement RASP (Runtime Application Self-Protection) to detect tampering. Configure reproducible builds in CI so independent parties can verify the published binary matches the source.
